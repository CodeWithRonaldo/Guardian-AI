⚙ Technical — Judges who know Sui / Move
Q What stops the agent from doing more than it's supposed to?

Move's type system. The agent can only call functions that take &GuardianCap as a parameter. Those functions are explicitly written by the protocol team — nobody else can add them. The blockchain rejects any other call at the VM level, not in application logic. There is no way to bypass it in code.

Q What if GuardianAI gets hacked or the agent wallet is compromised?

The protocol team holds an AdminCap. One transaction from their wallet calls cap::disable() and the agent loses all permissions instantly.

Even in a worst-case compromise, the damage is limited to whatever the GuardianCap was scoped to — which is only pause and tighten LTV. The agent cannot withdraw funds, cannot change ownership, cannot do anything outside that narrow scope.

Q Can the agent stop a flash loan attack?

No — and we don't claim it can. Flash loans happen within a single atomic transaction. By the time any off-chain engine sees it, it has already settled on-chain.

What the agent stops is multi-transaction attacks and the aftermath of single-transaction exploits — like a pool balance that craters after an exploit, or a price oracle that gets manipulated across several blocks. The Cetus exploit involved ongoing drain activity that an autonomous agent could have halted within seconds.

Q Why not use on-chain automation instead of an off-chain engine?

Pyth price feeds, TWAP calculations, and anomaly detection require comparing data over time from external APIs. You cannot run a loop that polls Pyth's Hermes API from inside a Move contract.

The off-chain engine is what gives the system real-world signal awareness. The on-chain part only executes — it never decides. That separation is intentional and is how all serious DeFi monitoring infrastructure works.

Q What data sources does the risk engine use?

Three sources, all verifiable:

1. Pyth Hermes API — SUI/USD price feed, polled every 3 seconds. Checks spot price deviation from a rolling TWAP and oracle publish timestamp staleness.

2. Sui RPC object polling — reads the Protocol shared object every 4 seconds for pool balance, LTV ratio, and paused state.

3. On-chain ActionLog — shared object that records every guardian action permanently. Walrus stores the full signal context for each entry.

◈ Product — Judges focused on real-world value
Q What's the real difference from Hypernative or Chainalysis?

They detect and alert. A human still has to see the alert, understand it, decide what to do, and act. That chain takes minutes at best.

GuardianAI closes the loop. The response is autonomous and on-chain. No human in the critical window. That is the missing piece — not better detection, faster response.

"Hypernative saw the Cetus exploit. They sent an alert. $60M moved to Ethereum before humans could coordinate a response. GuardianAI would have paused the protocol in the same 4-second window the attacker was operating in."
Q What protocols would actually use this?

Any Sui DeFi protocol that holds user funds — lending protocols like Suilend and Navi, DEXs like Cetus and Bluefin.

The proof point is the Cetus exploit (October 2025) — $60M drained before humans could respond. Every protocol team on Sui watched that happen and thought "that could be us." GuardianAI is the answer to that fear.

Q How do protocol teams trust you with this capability?

They don't have to trust us broadly. Three reasons:

1. Narrow scope. They define exactly which functions the agent can call. If they only expose pause_swaps, that's all it can do.

2. Full control. They keep the AdminCap. Revoke in one transaction, any time, for any reason.

3. Full transparency. Every action is permanently logged on-chain with the reasoning. Full signal context on Walrus. No black box.

Q Why does this need to be on Sui specifically?

Two reasons that are fundamental to the design, not marketing:

Move capability objects. On Ethereum, giving an AI agent on-chain permissions requires proxy contracts, multisigs, and governance delays — each adding its own attack surface. On Sui, a capability object is enforced by the VM's type system. Clean, composable, and safe by construction.

Sub-second finality. Sui confirms transactions in under a second. The agent can detect and respond within the same few-second window an attacker is operating in. On Ethereum with 12-second block times, that window is gone before your transaction even lands.

⚠ The Two Hardest Questions
Q What if the risk score is wrong and the agent pauses a healthy protocol?

Two answers, and both matter:

First — false positives are recoverable. The protocol team can unpause in one click using their AdminCap. A brief incorrect pause costs users a few minutes of downtime. An undetected exploit costs them everything. The asymmetry is clear.

Second — the scoring is designed conservatively. A single minor signal doesn't cross the pause threshold (score 85). Multiple signals have to fire together. A price deviation alone scores 30. Oracle staleness alone scores 20. It takes a combination of anomalies — the pattern of a real attack — to trigger an autonomous pause.

The frame to use: "Would you rather have a protocol that occasionally pauses for 2 minutes on a false positive, or one that stays open during a real exploit and loses $60M?"

Q This is just rule-based logic — where is the AI?

The risk scoring engine is rule-based — we say that clearly and don't hide it. The "AI" in GuardianAI refers to the autonomous decision-making loop, not a trained model.

The honest answer: a well-calibrated rule-based system that fires in 4 seconds is more valuable than an ML model that takes 30 seconds to produce a probability. Speed is the entire point.

The architecture is designed so that a real ML model — anomaly detection, pattern recognition trained on historical exploit signatures — can replace the scoring engine without changing anything else. The interface is the same. That's the roadmap.

Don't apologise for the rule-based approach. Own it. "We chose correctness and speed over complexity. The rules are transparent, auditable, and tunable by the protocol team. That's a feature."
