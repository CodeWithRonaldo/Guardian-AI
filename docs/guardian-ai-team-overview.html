01 · The Problem

DeFi moves faster than humans can respond

DeFi (Decentralised Finance) is financial software that runs on blockchains. Protocols like lending platforms and decentralised exchanges hold real money — sometimes hundreds of millions of dollars — managed entirely by code.

Beginner note — What is DeFi?

Think of DeFi as a bank, but one where there are no employees — just software running on a blockchain. Users deposit money, borrow against it, trade assets, and earn yield — all automatically, without a bank approving anything. Because there are no employees, if something goes wrong, there is no human to press the emergency button. That is the problem GuardianAI solves.

The Cetus Exploit — a real example

In October 2025, Cetus — Sui's largest decentralised exchange — was attacked. A bug in the code allowed an attacker to manipulate price calculations and drain liquidity pools. Around $60 million was moved to Ethereum before anyone could stop it.

The problem was not that nobody was watching. Monitoring tools like Hypernative exist and they detected the attack. They sent an alert.

But then the human response chain kicked in:

1

Alert fires

Monitoring tool detects the anomaly and sends a notification.

2

A human sees it

Someone has to notice the alert — could be minutes if they are asleep or busy.

3

A human understands it

They have to read it, understand what is happening, and decide what to do.

4

A human acts

They have to execute the response — coordinate with the team, sign a transaction, pause the protocol.

That entire chain takes minutes at best. The Cetus attacker operated in seconds. By the time humans coordinated, the money was gone.

The missing piece is not better detection. It is faster response. Specifically, autonomous on-chain response that does not wait for a human. That is what GuardianAI is.

02 · The Solution

What GuardianAI actually does

GuardianAI is a system that sits alongside any DeFi protocol and does three things continuously — every 3 to 4 seconds, around the clock:

W

Watches

Reads live price feeds from Pyth (a blockchain oracle service), reads on-chain protocol state (pool balances, LTV ratios, paused flags), and tracks oracle freshness. All of this happens in the background continuously.

T

Thinks

A risk engine scores the current situation from 0 to 100. It combines multiple signals — price anomalies, pool drain speed, oracle staleness — into a single risk score that updates every few seconds.

A

Acts

When the score crosses a threshold, the system does not send an alert — it executes a protective action directly on the blockchain. In under 4 seconds. No human required.

Every action is logged permanently on-chain. The protocol team can see what happened, why, and override it with one click. The human is still in control of the rules. The agent enforces them at machine speed.

04 · Smart Contracts

The Move smart contracts

Beginner note — What is a smart contract?

A smart contract is a program that lives on the blockchain. It is code that runs automatically when triggered, and nobody can change it once deployed. Think of it as a vending machine — you put in the right input, it gives you the right output, and there is no human deciding anything in between. Move is the programming language Sui uses to write smart contracts.

Beginner note — What is Sui?

Sui is a blockchain — a decentralised database that records transactions. It is faster than most blockchains (transactions confirm in under a second) and uses a programming language called Move which has strong safety guarantees built in. GuardianAI is built on Sui specifically because Move's type system makes the permission model safe in a way that is not possible on other blockchains.

The GuardianAI contracts are deployed at package address: 0x50b1a715...ac9b2d on Sui testnet. There are three modules:

🔑
guardian_ai::cap — The Permission System
File: contracts/guardian_ai/sources/cap.move

This module defines the two capability objects that control who can do what. A capability object in Sui is like a physical key — if you hold the key, you have permission. If you do not hold it, the blockchain rejects your transaction.

ObjectWho holds itWhat it allows
GuardianCap The agent wallet (backend) Call circuit breaker functions — pause protocol, tighten LTV
AdminCap Protocol team wallet Enable/disable the guardian, unpause, reset LTV. Full override.
GuardianConfig Shared — anyone can read Stores whether the guardian is enabled. Circuit breakers check this.

Why this matters: Move's type system enforces these permissions at the Virtual Machine level. You cannot call a function that requires &GuardianCap unless you literally own a GuardianCap object in your wallet. Not a password. Not an API key. A cryptographic object the blockchain itself verifies. This is the security foundation of the entire system.

📋
guardian_ai::action_log — The Audit Trail
File: contracts/guardian_ai/sources/action_log.move

This module defines a shared object called ActionLog that records every single action the guardian takes. It is append-only — entries can be added but never deleted. Anyone — the protocol team, their users, auditors, journalists — can read the full history at any time.

Each log entry contains: timestamp, risk score at the time, what action was taken (as a number code), and a plain English reason. The full signal context is stored separately on Walrus to keep on-chain gas costs low.

Object ID on testnet: 0xb48afb54...f827

🧪
guardian_ai::test_protocol — The Demo Protocol
File: contracts/guardian_ai/sources/test_protocol.move

This is a mock DeFi protocol used for testing and the hackathon demo. It represents the code a real protocol team would add to their own contract when integrating GuardianAI. In production, this code would live in the protocol's own package.

It has four important functions:

  • pause_protocol — requires GuardianCap. Halts all activity.
  • tighten_ltv — requires GuardianCap. Reduces the LTV ratio.
  • unpause_protocol — requires AdminCap. Resumes activity.
  • simulate_pool_drain — requires AdminCap. For demo only — drains the pool balance to trigger risk signals.

Object ID on testnet: 0x53096d53...93

05 · Backend Risk Engine

The Node.js risk engine

The backend is a Node.js service running on Render (a cloud hosting platform). It is the brain of the system — it never stops running, never sleeps, and makes all the risk decisions. It is split into several files, each with a single clear job.

Beginner note — What is an oracle?

A blockchain on its own does not know what the price of SUI is. It only knows what transactions have happened. An oracle is a service that brings real-world data (like prices) onto the blockchain. Pyth is the oracle we use — it publishes SUI/USD prices every few seconds on Sui. Our backend reads those prices off-chain via Pyth's API and uses them to calculate risk.

FileWhat it does
pyth.js Fetches SUI/USD price from Pyth's Hermes API every 3 seconds. Calculates a rolling TWAP (time-weighted average price) and measures how much the current price deviates from it. Also checks if the oracle data is stale (not updated in over 30 seconds).
chain.js Reads the Protocol shared object from Sui every 4 seconds via RPC. Gets pool balance, LTV ratio, and paused flag. Computes how much the pool balance has dropped since the last reading.
scorer.js The risk scoring engine. Combines price signals and chain signals into a single score from 0–100. Each signal has a weight. The final score is the weighted sum, capped at 100.
executor.js When told to act, builds and signs a Programmable Transaction Block (PTB) using the agent wallet's private key. Submits it to the Sui RPC. Handles the pause and tighten_ltv actions.
walrus.js After every on-chain action, stores a full JSON audit record to Walrus — the decentralised storage network. Contains raw signal values, price data, and the transaction digest.
index.js The main entry point. Runs the polling loops, calls the scorer, decides what action to take, and runs a small Express HTTP server for the dashboard to query.

The risk signals

The scorer combines seven signals. Multiple signals firing at the same time push the score higher — a single minor anomaly stays low, but the pattern of a real attack will spike the score.

SignalScore AddedWhat it means
Price deviates >10% from TWAP +30 Price is moving very fast — possible oracle manipulation
Price deviates 3–10% from TWAP +15 Price is moving unusually fast — worth watching
Oracle stale >30 seconds +20 Price feed stopped updating — possible oracle attack
Pool balance dropped >20% this interval +35 Large, sudden pool drain — classic exploit signature
Pool balance dropped 5–20% +15 Smaller drain — could be normal activity or early exploit
Pool balance <1% of baseline +90 Pool is almost completely drained — emergency, act immediately
Protocol already paused on-chain +10 Paused flag already set — informational signal

What happens at each score

ScoreActionOn-chain?
0 – 49Log only. Nothing visible happens.No
50 – 69Notify — sends a POST to the configured webhook URL (Discord, Slack, etc.).No
70 – 84Tighten LTV — submits a transaction that reduces the LTV ratio on-chain.Yes
85 – 100Pause Protocol — submits a transaction that halts all protocol activity.Yes
Beginner note — What is a PTB?

PTB stands for Programmable Transaction Block. On Sui, instead of sending one simple transaction at a time, you can bundle multiple operations together into one atomic block. Our executor builds one of these blocks, signs it with the agent wallet's private key, and sends it to Sui. If it is accepted, all operations execute together. If anything fails, none of them execute.

The HTTP API

The backend exposes two endpoints the dashboard uses:

EndpointMethodReturns

/config POST Updates thresholds, LTV tighten amount, and webhook URL in real time
06 · Frontend Dashboard

The React dashboard

The frontend is a React web application built with Vite. It uses the @mysten/dapp-kit library to connect Sui wallets and read on-chain data. It has four pages.

◈
Dashboard
The main monitoring screen

Shows the live risk score as a semicircular gauge (colour changes from green → orange → red as risk increases), the current SUI/USD price from the backend, on-chain protocol state (LTV ratio, pool balance, paused/active status), and a chart of the last 20 action log entries.

The gauge score comes from the backend's /status endpoint. The protocol stats come from reading the Protocol shared object on Sui directly, via the Sui RPC.

≡
Action Log
The permanent on-chain audit trail

Shows every action the guardian has ever taken, pulled directly from the ActionLog shared object on Sui. Each entry shows the timestamp, risk score at the time, the action taken (colour-coded), and the plain English reason.

This is not a backend database — it reads the blockchain directly. The data is permanent and cannot be deleted or edited by anyone, including GuardianAI.

⚙
Configuration
Guardian settings and on-chain permissions

Where the protocol team manages the guardian. Contains:

  • Guardian toggle — Enable or disable the guardian with one click. Requires the AdminCap wallet to be connected. Signs and submits an on-chain transaction.
  • Risk threshold sliders — Set the score at which each action fires. Saved to the backend via POST /config and to localStorage for persistence across page reloads.
  • LTV tighten amount — How many basis points (1 basis point = 0.01%) to reduce LTV when the tighten action fires.
  • Webhook URL — Paste a Discord, Slack, or any HTTP webhook. The backend posts an alert here when the risk score crosses the Notify threshold.
▶
Simulation
The demo weapon

Replays a 30-second Cetus-style exploit scenario entirely in the browser — no real transactions, no CLI required. The gauge climbs in real time, signals turn red, and the system shows which action would fire and when.

The scenario is based on the actual Cetus attack pattern: gradual price deviation followed by oracle staleness and rapid pool drain. The score climbs from 0 to 80+ over 30 seconds, triggering Notify then Tighten LTV actions with visible timestamps.

This is the primary demo tool for the hackathon presentation.

07 · Walrus Integration

What Walrus is and why we use it

Beginner note — What is Walrus?

Walrus is a decentralised storage network built by the Sui team. Instead of storing files on Amazon S3 or Google Drive (which are centralised and can be shut down or censored), Walrus stores data across a distributed network. Once stored, the data gets a unique ID (called a blob ID) and can be retrieved by anyone. It is one of the hackathon tracks, which is why we integrate it.

We use Walrus for two things:

1. Extended audit logs. The on-chain ActionLog keeps entries small to minimise gas costs — it stores the action code (a number) and a short reason string. After every autonomous action, the backend writes a full JSON blob to Walrus containing all the raw signal values, the Pyth price at the time, the pool balance, the transaction digest, and the timestamp. The Walrus blob ID is what links the compact on-chain entry to the full context.

2. Verifiable audit trail. Protocol teams and their users can point to a Walrus blob ID and independently verify what the guardian saw and why it acted. No one can modify or delete the blob after it is stored.

08 · Security Model

Why there are two wallets

The system uses two separate wallets deliberately. Each represents a different entity with a different level of trust.

WalletAddressHoldsCan do
Admin wallet
(Protocol team)		 0xdab28d...f390 AdminCap Enable/disable guardian, unpause protocol, reset LTV, full override
Agent wallet
(Backend)		 0xc56efa...1339 GuardianCap Pause protocol, tighten LTV. Nothing else.

The agent wallet's private key lives in the backend's environment variables on Render. It signs transactions automatically. But it can only sign transactions that call functions accepting &GuardianCap as a parameter — which are only the circuit breaker functions explicitly written by the protocol team.

If the backend were ever compromised, the attacker would have the agent wallet's key. They could pause the protocol or tighten LTV — both recoverable actions. They could not withdraw funds, change ownership, or do anything outside the GuardianCap's scope. The AdminCap wallet is completely separate and the backend never touches it.

Think of it like a security guard at a building. The guard has a key to lock the front door in an emergency — but not the master key to every room and the safe. Two keys, two levels of trust.
